What is it?

    It's a GUI for field crew collecting infrastructure survey data (manhole, pipe inspection etc.) Basic usages are Data Visualization and Collection. It's also supposed to be a helper for the user: auto connect sensors based on the project, monitoring sensors or actuators status and inform user any connection error during the survey. Due to company's privacy, I cannot share the code for the software but it's similiar to a personal project that I was working on here which is modified from a qt-ros template from ROS wiki which I found when I was interested in Qt integrated with ROS.

Strategy

    The software is designed to be ROS-based was because we would like to take advantage of a lot of ROS powerful library like rosbag, dynamic reconfigure, hardware manager and RVIZ. For example, rosbag is really handy to have when testing SLAM with different parameters. Well in our case, it's more of 3d construction instead of SLAM.

      1. Socket programming for sensor/actuator ROS drivers. This is due to the fact that the robot gonna travel for a long distance so we prefer ethernet connection. I will need to read manual and write either UDP or TCP socket when I. the sensor/actuator need to use a serial-ethernet module II. there is no ROS driver for the sensors/actuators. I used mostly Python Socket since it's much easier and fast to validate the ideas. But I will definitely need to read more C/C++ for it since performance wise, C/C++ is much better. Here is a good resource to learn about it. Networking is a complicated and interesting topic and I would definitely like to know more.

      2. RVIZ cpp library for data visualization Since it is well-supported for ROS topic mechanism and well-integrated with Qt.

      3. Camera streaming using Gstreamer pipeline and Qt Video Widget. Collection using OpenCV and FFMPEGQt Media Player support Gst pipeline I have also tried in C code but really long code for the same functionality of one command line. Collection side, we need subtitled the video with sensor reading I tried ros callback function and then add sensor readings to the image frame by frame. Also tried to use Gstreamer's textOverlay and make it dynamic with GstControlSource But I prefer the first approach since it is much more straight-forward.

      4. Other sensor data collection using csv writer (simply std::ofstream) and rosbag (backup data)the reason for having csv files is that it is easier for post-processing team

      5. Project database management with std::ifstream This is designed for the purpose of automate the collection job. eg. based on the project database, automatically connect specific sensor/actuators and load the survey side information eg. the survey site's length and diameters. A more advanced way to do it is to use SQL with SQL API. This is more secure and faster (csv is flat-file database). So for now the database is pretty simple but better switch to SQL before things get bigger.
